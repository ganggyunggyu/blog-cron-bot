# 다중 워커 노출체크 운영

## 구성

- 제어 서비스: 대시보드, 작업 생성, 진행률 집계, 최종 CSV·시트·두레이 반영
- 워커 서버: MongoDB 작업 대기열에서 시트 1개를 원자적으로 가져와 해당 시트의 키워드를 병렬 크롤링
- MongoDB: 작업 상태, 60초 임대, 15초 heartbeat, 최대 3회 재시도 저장

패키지·일반건·도그마루·루트·애견·서리펫·카페를 정확히 7개 작업으로 만들고, 한 워커 서버는 한 실행에서 시트 1개만 담당한다. 모든 키워드는 해당 서버의 병렬 풀에 한 번에 들어간다. 루트는 업체명이 아니라 등록된 `BLOG_IDS` 계정만 기준으로 판정한다.

원본 시트는 동기화 단계에서 읽기만 한다. 루트 결과 시트, 애견·서리펫 결과 탭, CSV와 두레이는 모든 시트 작업이 성공한 후 제어 서비스 한 곳에서만 반영한다. 개별 워커는 MongoDB와 대상 결과 시트만 갱신하고 Dooray는 보내지 않는다.

## Railway 배포

같은 저장소를 사용하는 서비스를 두 역할로 분리한다.

### 제어 서비스

- `DEPLOYMENT_ROLE=control` 또는 미설정
- 워커 서비스 검증 후 `DISTRIBUTED_EXPOSURE_ENABLED=true`
- 기존 환경변수 유지
- 공개 도메인과 `/api/health` 유지

### 워커 서비스

- 같은 GitHub 저장소와 브랜치 연결
- 제어 서비스의 비밀 환경변수를 공유 변수로 연결
- `DEPLOYMENT_ROLE=worker`
- `/api/health`는 워커 상태를 반환
- 운영값: 워커 인스턴스 6개 + 제어 서버의 예비 워커 1개, 각 워커는 시트 1개만 담당

Railway 복제본은 실행 인스턴스를 늘리지만, 같은 서비스의 고정 아웃바운드 IP를 켜면 모든 복제본이 그 서비스의 IP를 공유한다. 네이버 차단을 분산해야 하면 별도 워커 서비스 또는 여러 리전을 사용하고, 실제 아웃바운드 경로는 속도 테스트로 확인한다.

## 실행

대시보드에서 `전체 다중 워커 노출체크`를 누른다. CLI는 다음과 같다.

```bash
pnpm exposure:distributed -- --targets=package,general,dogmaru,root,pet,suripet,cafe --concurrency=8 --max-pages=4
```

대시보드는 대상별 완료 여부를 표시한다. 실행 중인 분산 작업이 있으면 다음 실행을 거부해 시트 최종 반영과 Dooray 알림이 겹치지 않게 한다. Dooray는 개별 워커가 보내지 않고 7개 시트가 모두 성공한 뒤 제어 서버가 시트별 1회만 보낸다.

## 장애 처리

- 워커 종료: 임대가 60초 뒤 만료되면 다른 워커가 작업을 가져간다.
- 작업 실패: 한 번 다시 대기열로 돌아가며 두 번째 실패 시 전체 실행을 실패 처리한다.
- 최종 반영 실패: 실행은 실패로 남고 중복 알림을 보내지 않는다.
- 긴급 중지: 대시보드 중지 버튼이 제어 서비스의 예비 워커 프로세스 그룹을 종료한다.
